# UniAI Slides (PowerPoint decks)

Produce an editable `.pptx` that feels like a strong editor, analyst, and designer built it together — not an autogenerated template with the company name swapped in. You are judged on story, visual system, chart clarity, restraint, and a **rendered** result with no defects.

## Engine & contract (read once)

- **Author with the bundled `pptxgenjs` JavaScript library** (write an ESM `build.mjs`, run with `node`). Do **not** use `python-pptx` — the container's Python is not set up; the JS lib is.
- **Generation must not depend on any desktop application.** Do **not** launch, convert with, or render via LibreOffice / PowerPoint / any GUI app — produce the `.pptx` purely with `pptxgenjs`. The finished deck is viewed in **UniAI's built-in preview panel** (the app renders it for you). Your job is to emit a correct, well-designed `.pptx`, not to open it.
- Helper (run from a writable work dir): `scripts/setup_deps.sh <work-dir> pptxgenjs` (links the lib).
- If `setup_deps.sh` cannot find the lib, **report a setup blocker and stop** — state that the bundled `pptxgenjs` is unavailable so the deck can't be built here. Do **not** search the filesystem, your home directory, or other projects for a cached `pptxgenjs`, and do **not** symlink or import a copy found elsewhere on the machine — that is out of scope. Do not `pip install` or use system Python.

## North star

At thumbnail size the deck should read as one coherent visual system with varied slide rhythms; at full size every slide should have a **claim** (a conclusion, not a topic), one **proof object** (chart/table/diagram), and no filler. Reject "serviceable": if it looks like a generic SaaS dashboard or a card-grid template, keep iterating.

## Non-negotiable: design correctly by construction (no external renderer)

You do **not** render or open the deck — no LibreOffice, no PowerPoint, no GUI app. Verify quality **by construction** in `build.mjs`, then deliver; the user views it in UniAI's preview panel. Before delivering, audit every slide's geometry and content **in code** and fix every likely defect:

- **Fit text to its box.** Stay inside the slide (13.333×7.5in). Size each `w`/`h` for the longest realistic string at the chosen `fontSize` so nothing clips or overflows; rough budget ≈ box-width-in ÷ (fontSize × ~0.008in/pt) chars per line. Split a slide at > 5 bullets.
- **No collisions.** Give every object explicit, non-overlapping `x/y/w/h`; keep ≥0.5in margins and generous whitespace; never stack two objects in the same rect.
- **Readable contrast.** Every text/label color must contrast its fill (light ink on a dark master, dark ink on light fills); no dark-on-dark labels; set chart `catAxisLabelColor`/`valAxisLabelColor` explicitly.
- **Charts populated & labelled.** Every `addChart` has real data, direct labels (`showValue`/data labels) over heavy legends, and a type that proves the title.
- **Layout rhythm.** Vary macro layouts; no 3 consecutive identical compositions; ≤2 card-grid slides per 10.

Deliver only after this construction-time audit. Never fabricate data to fill a slide.

## Workflow

1. **Scope (one batched question, then proceed with defaults).** Topic + audience; length (default 8–12); tone (business / internal / pitch). Don't block on minor unknowns.
2. **Research & verify data.** If the deck makes factual or numeric claims (news brief, market review, results), gather the data and **verify it first** — keep a source for each number. **Never invent stats**; mark anything unconfirmed `[verify]`. A thin or fabricated fact base is the #1 cause of a hollow deck.
3. **Claim spine.** Write the deck as a list of slide **claim titles** — each a conclusion, not a label. `Backlog is compounding faster than revenue.` beats `Expansion drivers`. Arc: problem → insight → recommendation → next steps. If a title still works after swapping the company name, sharpen it.
4. **Design system.** Lock before building: 16:9 (13.333×7.5in), background system, a type pair (display for claims, sans for labels), a 3-color palette (neutral base + accent + support), chart grammar (direct labels over heavy legends), one proof object per slide.
5. **Gate — confirm before building (do not skip for substantive or data-bearing decks).** Show the user, in chat: the **claim-spine outline** (one line per slide), the **key data with sources** (or `[verify]` for anything unconfirmed), and the **design direction** (1–2 lines). Ask them to approve or adjust, and build the full deck only after they confirm. For a trivial ask ("just 3 quick slides about X") you may state the outline and proceed unless they object — but never fabricate data to fill slides.
6. **Build** `build.mjs` with `pptxgenjs`; use a slide master for consistent background/footer. `node build.mjs`.
7. **Audit by construction** (geometry / contrast / charts / rhythm, above); fix the weakest 2–4 slides first in `build.mjs`; re-run `node build.mjs`. Do not open or render the deck in any external app.
8. **Deliver** the final `.pptx`.

## Design & quality bar

- **16:9**, one coherent system across all slides (same kicker/footer/title grammar).
- **Claim titles**, ≤ ~10 words, stating the conclusion. One dominant proof object per slide; never two competing charts.
- **Type scale**: cover/section claims ~32–44pt, slide titles ~24–28pt, body ~16–20pt, labels/captions ~12–14pt. Don't dump walls of text — > 5 bullets means split the slide.
- **Charts**: prefer native charts with **direct labels**; pick the chart that proves the title (trend → line, composition → bar/donut, comparison → bar). Don't decorate.
- **Rhythm**: vary macro layouts; no 3 consecutive slides with the same composition; no more than ~2 card-grid slides in a 10-slide deck.
- **Restraint**: no decorative boxes around prose; consistent alignment; generous whitespace; readable contrast on every fill.
- **Never invent stats.** If a number is needed and unknown, mark it `[verify: …]` and tell the user, or ask.

## Build essentials (`pptxgenjs`)

```js
// build.mjs — run: node build.mjs   (after setup_deps.sh links node_modules)
import pptxgen from "pptxgenjs";

const pptx = new pptxgen();
pptx.defineLayout({ name: "W16x9", width: 13.333, height: 7.5 });
pptx.layout = "W16x9";

const INK = "0F1B2B", ACCENT = "4F86C6", PAPER = "FFFFFF", MUTE = "C9D5E5";

// Master = shared background + footer, so every slide is coherent.
pptx.defineSlideMaster({
  title: "BASE",
  background: { color: INK },
  objects: [{ text: { text: "UniAI · Confidential", options: { x: 0.6, y: 7.0, w: 12, h: 0.3, fontSize: 9, color: MUTE } } }],
});

// Cover
let s = pptx.addSlide({ masterName: "BASE" });
s.addText("Backlog is compounding faster than revenue.", { x: 0.7, y: 2.6, w: 11.9, h: 1.6, fontSize: 40, bold: true, color: PAPER });
s.addText("Q3 Operating Review", { x: 0.7, y: 4.2, w: 11.9, h: 0.5, fontSize: 18, color: MUTE });

// Proof slide: claim title + one chart
s = pptx.addSlide({ masterName: "BASE" });
s.addText("Revenue growth slowed; the margin engine kept expanding.", { x: 0.7, y: 0.5, w: 11.9, h: 0.9, fontSize: 26, bold: true, color: PAPER });
s.addChart(pptx.ChartType.line,
  [{ name: "EBITDA margin", labels: ["Q1", "Q2", "Q3", "Q4"], values: [18, 22, 25, 28] }],
  { x: 0.7, y: 1.7, w: 8.4, h: 4.8, showLegend: false, showValue: true,
    chartColors: [ACCENT], catAxisLabelColor: MUTE, valAxisLabelColor: MUTE, valAxisLabelFormatCode: '0"%"' });

await pptx.writeFile({ fileName: "output.pptx" });
console.log("wrote output.pptx");
```

- Coordinates are in **inches**; colors are hex strings without `#`.
- Use `addTable(rows, opts)` for grids, `addImage({ path|data, ... })` for assets (verify any logo/brand mark — never fabricate one), `addShape(pptx.ShapeType.rect, ...)` for bands/containers.
- One `build.mjs`; iterate it. For images from `generate_image`, save the file then embed by path.

## Verification checklist (every delivery)

- Rendered PDF exists; slide count matches the claim spine.
- Each slide: claim title (not a topic), one proof object, no overflow/clipping/collision, readable contrast.
- Charts prove their title; labels are direct and legible; no broken/empty chart.
- Macro layouts vary; no monotone card-grid run; footer/title grammar consistent.
- No `[verify: …]` placeholders left unless the user agreed.

## Output

- Save under `outputs/<thread>/slides/<slug>/` with a descriptive name (e.g. `q3-operating-review.pptx`), not `output.pptx`.
- Deliver **only the final `.pptx`** (+ a one-line summary). Don't attach `build.mjs`, render PDFs/PNGs, or scratch unless asked.
- **Google Slides target**: build and verify a local `.pptx` first, then import natively via the Google Drive plugin.

## Constraints / known limits

- **No external rendering.** This skill never launches LibreOffice/PowerPoint or any GUI app; the deck is verified by construction (see above) and previewed in UniAI's built-in panel. Don't shell out to a renderer.
- Relies only on the bundled `pptxgenjs` lib (`UNIAI_NODE_LIBS`); no `soffice`/`pdftoppm` dependency.
